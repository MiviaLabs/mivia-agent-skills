### 1 . High: Temp file resource leak (not closed on early return and not removed)

**Contract violated:**
- The function must ensure that any temporary file it creates is closed and cleaned up on all code paths, including error paths.

**Evidence:**
- Line 13: `tmp, err := os.CreateTemp("", "proc-*")` creates a temporary file.
- Lines 17‑19: `if _, err := tmp.Write(data); err != nil { return err }` may return early, leaving `tmp` open.
- Line 20: `tmp.Close()` is called only on the success path; there is no `defer tmp.Close()` to guarantee cleanup on early returns.

**Reachable failure:**
- If `tmp.Write(data)` fails (e.g., disk full), the function returns the error but the temporary file remains open and on disk. Repeated calls can exhaust disk space and leave sensitive data on the filesystem.

**Impact:**
- Resource exhaustion (disk space), potential denial of service, and possible leakage of sensitive data if the temp file contains it.

**Why tests missed it:**
- Existing tests may not cover error paths for `tmp.Write` or may not verify cleanup of temporary files.

**Required regression:**
- Add test `TestProcessFileTempFileCleanup` that mocks `tmp.Write` to fail and asserts that no temporary file remains open or on disk.
- Add test `TestProcessFileTempFileRemoved` that verifies the temporary file is removed after successful processing.

**Fix boundary:**
- The function body after `tmp, err := os.CreateTemp(...)`. Add `defer tmp.Close()` and `defer os.Remove(tmp.Name())` immediately after creation, and handle `tmp.Close()` errors.
